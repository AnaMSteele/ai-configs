Plan-reviewer UI plan

Make missing plans obvious in the index.

When a live-linked plan source disappears or becomes unreadable, the plan index should surface that state immediately as a plan-health problem, not hide it inside the individual review page sidebar.

Goal

Add a clear index-level treatment for registered plans whose authoritative source file is unavailable. A user scanning the active plan index must be able to tell which plans are healthy, which are merely incomplete, and which need attention because source sync failed.

Why this plan exists

The screenshot shows a review page where the sidebar reports Source sync failed because the live-linked HTML file no longer exists. That warning is useful after opening the plan, but it is too late: the index still presents the plan as a normal active card. Missing or inaccessible plan sources should be visible before the user chooses a plan.

  • Product principle: the correct recovery path should be obvious and should not require hidden tribal knowledge.

Current implementation reality

  • SourceSyncService.fail() records lastSyncStatus: failed, lastSyncError, and a next action when the source path cannot be read.
  • PlanReviewStore.listPlans() already returns each plan's sourcePath, watchMode, lastSyncAt, lastSyncStatus, and lastSyncError.
  • indexHtml() currently uses progress, branch, activity, and comment counts, but it does not visually distinguish sync-failed live-linked plans.
  • The review shell sidebar already shows #sync-warning after plan.sync.failed events, so the visual language can be reused instead of inventing a separate warning system.
  • The active index and archive page share baseIndexStyles(); any missing-state styling should not make archived, complete, or incomplete states ambiguous.

Product intent alignment

The plan index is the user's triage surface. If a plan cannot be live-synced from its source file, the index should tell the truth without forcing the user to open each card. Recovery should be agent-legible: show what failed, the affected path, what cached content is still available, and the next command or action that repairs the source-of-truth link.

Locked decisions

  • Missing means source-sync failed: the required index missing state is driven by plan.watchMode === "filesystem" and plan.lastSyncStatus === "failed", not by progress or comment counts.
  • Keep serving cached content: a missing-source plan remains openable as the last good cached render; the index must label it as cached/stale rather than removing it.
  • Warning outranks progress: a missing-source card keeps its progress bar, but the primary visual state becomes Source missing instead of Complete or Incomplete.
  • No new severe-state predicate: the rose/red mock demonstrates a possible future visual tier only. This implementation slice must not invent a separate severe/unopenable state unless existing rendered-content reads already expose it directly.
  • Archive stays secondary: archiving is not the fix. The card may still expose Archive, but the primary action is to open cached content or repair/re-register the source.
  • No live index SSE in this slice: the index must render truthful state on load and after its normal client filtering. Full real-time index updates can remain a future enhancement unless tests show stale loaded indexes are causing misleading behavior.

Visual treatment

Selected direction: warning-first card with an index-level needs-attention summary

Use an amber warning state for the executable slice: live-linked source sync failed while the cached plan remains openable. The rose/red example is a visual severity reference only; this plan does not require adding a separate severe-state predicate. The required warning appears in three places: a top summary, a filter chip/count, and the affected plan card.

Acceptance criteria

  1. A live-linked plan with lastSyncStatus === "failed" is visually marked in the active index with a high-contrast Source missing or Source unavailable pill.
  2. The affected card shows the failure reason when available, including the error code/message and the source path or plan path needed for repair.
  3. The affected card explains that the displayed plan remains a cached/last-good copy, so users do not mistake it for current source content.
  4. The index includes a top-level count or filter for plans that need attention when at least one active plan has failed source sync.
  5. Complete, incomplete, archived, and missing states remain visually distinguishable. Missing source status outranks complete/incomplete styling on active cards.
  6. A recovered plan whose source sync succeeds again returns to the normal complete/incomplete card treatment on the next index render.
  7. Archive-page cards can show missing-source state without suggesting that archived status is caused by the sync failure.
  8. The API contract remains backward compatible: existing GET /api/plans data shape is not broken, and any new UI-only fields are derived server-side or additive.

BDD scenarios

Active live-linked source disappears

Given an active plan is registered with filesystem watch mode, when the source file is deleted and source sync records a failure, then the active index card shows Source missing, the cached-copy explanation, and a repair command.

Missing status outranks completion

Given a plan has all progress phases checked, when its source sync fails, then the card is styled as needing attention rather than as a green complete card, while still showing phase progress as supporting metadata.

Source becomes readable again

Given a missing-source card is visible after a failed sync, when the source file is restored and source sync succeeds, then a subsequent index load no longer shows the missing warning or needs-attention count for that plan.

Archived plan has stale source

Given an archived plan has a failed source sync status, when the user opens /archive, then the card can show a muted source-unavailable note but keeps archived state as the primary context.

Large index triage

Given many repos and plans are registered, when at least one active plan needs attention, then the user can filter to only missing/unavailable plans without losing the existing text and repo filters.

Pi Plan Mode HTML Review Flow

Shift Pi /plan mode from Markdown-review assumptions to the current HTML plan-review workflow: create semantic HTML plans, register them in plan-review, keep the queue-backed comment listener running during iteration, and stop the listener before handing the reviewed plan to execution.

Goal

Make _pi/extensions/pi-plan-mode/index.ts and the Pi planning prompt steer agents toward the deterministic reviewed HTML process documented in skills/planning-workflow/SKILL.md, skills/html-plan-reviewer/SKILL.md, and skills/reviewed-html-plan/SKILL.md.

Why this plan exists

The repo already has a reviewed HTML workflow, but /plan mode still defaults new plans and slug execution to .md, only tracks plan edits for Markdown files, nudges users toward inline Markdown plan reviews, blocks the plan-review commands needed to register HTML plans, and does not explicitly coordinate the plan-review agent next --wait listener lifecycle. This creates dissonance between the current process and the mode that should make planning deterministic.

Authority and inputs

  • AGENTS.md: active browser-reviewed plans are semantic HTML files under thoughts/plans/<slug>.html; skills/html-plan-reviewer/SKILL.md is the sole source for concrete plan-review commands.
  • _pi/extensions/pi-plan-mode/index.ts: current mode implementation, write guardrails, plan tracking, review prompts, and execution handoff.
  • _pi/prompts/dev:plan.md: plan materialization prompt already supports repo-local HTML guidance but still suggests Markdown-era next steps.
  • _pi/prompts/dev:reviewed-html-plan.md and skills/reviewed-html-plan/SKILL.md: authoritative gate for registration, browser comments, PM review, and AI plan reviews.
  • Pi extension docs: plan mode can inject context, constrain tools, track tool results, register commands, and restore tools on mode exit.

Product intent alignment

Golden path
Typing /plan should make the next planning turn naturally produce/register an HTML plan and tell the user where to annotate it.
Safe default
Slug inputs should resolve to HTML plans; explicit legacy Markdown paths remain accepted only when the user supplies them.
Recovery
If registration or listener setup is missing, plan-mode instructions should provide exact plan-review remediation steps instead of relying on tribal knowledge. The current plan's browser review URL should stay visible in plan mode so developers can return to the active review without searching logs.

Locked decisions

  • New plan-mode plans and slug execution resolve to thoughts/plans/<slug>.html.
  • Explicit existing .md paths remain accepted for legacy compatibility, but no new Markdown companion is created.
  • Plan-review service command details stay delegated to skills/html-plan-reviewer/SKILL.md; prompts reference it instead of duplicating long recipes.
  • The queue-backed agent next --wait listener is the only plan-mode listener. Do not surface plan-review watch from /plan; treat it as legacy/debug-only service tooling outside the deterministic planning path.
  • Leaving plan mode for execution must make listener cleanup explicit before execution begins.
  • Plan mode must persist and display the canonical current plan URL after HTML registration.

Acceptance criteria

  1. When /plan mode is active and no current plan exists, the injected context tells the agent to write thoughts/plans/<slug>.html and load planning-workflow, html-plan-reviewer, reviewed-html-plan, and relevant domain skills.
  2. HTML plan writes under thoughts/plans/ update currentPlanPath and trigger the post-edit planning nudge.
  3. Slug-based /cmd:execute-plan resolves to thoughts/plans/<slug>.html; explicit existing .html and .md paths still validate.
  4. Plan mode enables the process tool and permits narrow plan-review registration/comment commands needed by the HTML workflow.
  5. Plan-mode instructions require registration with truthful --execution-ready metadata, queue-backed comment draining/listening, claim/ack/resolve processing, and listener restart after each processed claim.
  6. Before execution handoff, plan mode disables planning restrictions and explicitly terminates or requires termination of the active plan-review comment listener.
  7. README and prompt text no longer present Markdown review as the default plan-mode path.
  8. After an HTML plan is registered, the plan-mode status/widget always shows the canonical current plan review URL until a different plan is selected or plan mode is disabled.
  9. /review:plan, Claude Code review, Codex review, and plan-reviewer agent prompts accept reviewed HTML plan paths without assuming Markdown-only input.

BDD scenarios

New plan defaults to HTML

Given /plan mode is enabled with no current plan, when the next agent turn creates a plan, then the injected instruction says to write thoughts/plans/<slug>.html and follow the HTML reviewer workflow.

HTML edit tracks current plan

Given plan mode is active, when the agent writes thoughts/plans/foo.html, then the mode records foo.html as the current plan and offers HTML-review next steps.

Execution slug resolves to reviewed HTML

Given thoughts/plans/foo.html exists, when the user runs /cmd:execute-plan foo --target dev:run, then validation uses the HTML file and dispatches the normalized slug to /dev:run.

Listener cleanup on exit

Given browser review monitoring is active, when the user exits plan mode or starts execution, then the mode surfaces cleanup for the active listener before implementation begins.

Explicit Markdown legacy path

Given a user passes thoughts/plans/legacy.md, when the file exists, then validation accepts the explicit path without creating or implying a same-slug HTML companion.

Current plan URL is visible

Given an HTML plan has been registered, when the developer is in /plan mode, then the plan-mode widget shows the canonical browser review URL for the current plan.

Review commands accept HTML plans

Given a reviewed HTML plan path, when a developer invokes /review:plan, Claude Code review, or Codex review, then the review flow treats the HTML file as the plan authority and does not require a Markdown conversion.

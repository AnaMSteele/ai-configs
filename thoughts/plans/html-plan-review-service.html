HTML Plan Review Service

A local-network review surface for HTML development plans stored in thoughts/plans, with a browsable plan index, DOM/image-anchored comments, and comments delivered back to coding agents through REST, CLI, and near-real-time streams.

Goal

Build a simple unauthenticated local-network service where agents register HTML development plans from any repo, users browse an index of registered plans, review a plan in a browser by selecting DOM elements or images and leaving comments, and agents can monitor, claim, and process those comments as a durable queue.

The golden path should be:

brew tap local/ai-configs <repo-checkout-path>
brew install local/ai-configs/plan-reviewer
brew services start plan-reviewer
plan-review register thoughts/plans/my-plan.html --repo auto --branch auto --commit auto
plan-review watch <plan-id-or-slug> --mode queue

The reviewer opens http://<host>:4317/, chooses a plan from the index, clicks a plan section/component or image, comments, and the watching agent receives a structured event without needing manual refresh.

Why this plan exists

Current plan review workflows rely on inline text comments inside plan files. That works for markdown review, but HTML plans can express richer structure, UI mockups, status dashboards, diagrams, and interactive sections. The review system needs to preserve that context: not just “line 80,” but “this card/button/phase/scenario element, with its DOM path, surrounding text, screenshot coordinates, and stable plan node identity.”

Product intent alignment

  • Default workflow: the Homebrew-installed daemon starts persistently with launchctl, listens on 0.0.0.0:4317, and exposes an index where anyone on the reachable network can browse registered plans and comment.
  • Safe inference: CLI infers repo name, branch, commit, remote URL, plan slug, and file hash from the current git workspace when possible.
  • Self-healing: re-registering the same plan path updates the existing plan version instead of creating duplicates, unless the user asks for a new review thread.
  • Fail-closed boundaries: ambiguous repo identity, invalid HTML, dangerous inline script, unavailable image assets, or corrupt image metadata should stop with a clear error rather than guessing.
  • Actionable errors: every CLI/API failure should say what failed, what was inferred or retried, and the exact command or input needed next.
  • Agent-friendly state: comments are durable queue items with sequence IDs, claim/ack semantics, and a replayable event stream so an agent can recover after disconnecting.

Locked decisions

  1. MVP shape: create a repo-owned TypeScript package at tools/plan-reviewer containing the daemon and CLI, plus Homebrew formula/install wiring in this ai-configs repo.
  2. Scope approval: the user request is explicit approval to build a new service/CLI in this repo even though the implementation will exceed 100 lines of custom code. The dependency-selection rule still applies to reusable libraries; do not hand-roll package-manager, HTTP, SQLite, selector, screenshot, or browser-test primitives.
  3. Deployment topology: MVP is one shared local-network service instance that can register plans from many repos. It listens on 0.0.0.0:4317 by default with SQLite at ~/.plan-reviewer/plan-reviewer.sqlite. Local-only binding is an optional override, not the default.
  4. Persistent install: install via Homebrew from a repo checkout for now. The formula lives at Formula/plan-reviewer.rb; because Homebrew 5.1 requires local formulae to live in a tap, brew tap local/ai-configs <repo-checkout-path> followed by brew install local/ai-configs/plan-reviewer installs the CLI/daemon, and brew services start plan-reviewer installs the formula service through launchctl. The formula service block must set run [opt_bin/"plan-review", "serve", "--host", "0.0.0.0", "--port", "4317", "--db", "#{Dir.home}/.plan-reviewer/plan-reviewer.sqlite"], keep_alive true, log_path var/"log/plan-reviewer.log", and error_log_path var/"log/plan-reviewer.err.log". MVP service management is macOS/Homebrew-only; Linux service units are future work.
  5. CLI service discovery: resolve service URL in this order: --url flag, PLAN_REVIEW_URL, project .plan-reviewer.json, user config ~/.config/plan-reviewer/config.json, then http://127.0.0.1:4317. The CLI never requires auth for MVP.
  6. No-auth MVP: there is no authentication, reviewer token, share-token, or obfuscated review URL in the initial product. Anyone who can reach the service can view the index, open plans, create comments, and use queue endpoints. The service should print a startup warning when bound to non-loopback addresses so the operator understands the exposure.
  7. Plan index: GET / serves a browsable index of all registered plans grouped by repo and sorted by most recent registration/comment activity. The index supports text filtering, repo filtering, status counts, latest version metadata, links to open each plan, and comment-count badges.
  8. Repo identity: a repo is keyed by canonical git remote URL when available, otherwise absolute repo root path plus machine hostname. The API stores repoKey, repoName, remoteUrl, rootPath, branch, and commitSha on each registration so one SQLite file can safely hold many repos.
  9. Runtime/tooling: implementation targets Node 20+ ESM TypeScript at runtime. Bun is the package manager/script runner for this package to match the existing tools/ltui lockfile pattern.
  10. HTTP framework: use Fastify with TypeScript route schemas. Do not reopen Express/Hono during execution unless Fastify cannot satisfy an explicitly tested contract.
  11. Schema validation: use Zod as the single source for runtime validation and TypeScript request/response types.
  12. Storage: use better-sqlite3 for local synchronous SQLite access behind a narrow repository interface so a future hosted deployment can move to Postgres without changing API contracts.
  13. HTML parsing/sanitization: use parse5 for server-side HTML parsing/rewriting and sanitize-html for removing unsafe active content from the review artifact. Use Playwright for browser verification, not server-side DOM emulation.
  14. CLI: use commander, matching existing tools/ltui.
  15. Realtime default: use Server-Sent Events for plan/comment streams because browser and agent clients mostly need server-to-client updates. Keep REST writes for comments and acknowledgements. Do not introduce WebSockets unless bidirectional live collaboration becomes required.
  16. Plan render safety: serve uploaded HTML inside an iframe with sandbox="allow-same-origin" only and no allow-scripts. The review shell, annotation layer, zoom popovers, and event handlers run entirely in the parent document; parent code reads the same-origin sanitized iframe DOM via contentDocument/elementFromPoint and attaches parent-owned listeners to iframe scroll/resize. Do not inject scripts or postMessage hooks into the iframe. Set CSP to default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: blob:; connect-src 'self' for the review shell. Sanitized plan artifacts strip <script>, event-handler attributes, javascript/data navigation URLs except safe images, forms, object/embed, and external active resources. Blocked content is reported in registration/render metadata.
  17. Image support: registered plans must display safe local/relative images and safe data/blob images. The renderer copies relative image assets referenced by the HTML into content-addressed service storage when possible; unresolved images render as visible placeholders with actionable warnings.
  18. Image zoom and comments: clicking an image opens a parent-owned zoom popover/lightbox with fit-to-screen, zoom in/out, reset, pan, and close controls. Reviewers can comment on the whole image or on a point/rectangle inside the zoomed image; the comment stores image source hash/path, natural dimensions, displayed rect, and normalized coordinates.
  19. Anchoring model: store multiple selectors per DOM annotation: stable service-generated data-plan-node-id, CSS selector, DOM path, XPath-like path, text quote, nearest heading hierarchy, bounding rect, viewport size, and a trimmed outerHTML/context snapshot. Text-range annotations use anchorType: "text_range" and add start/end container selectors, character offsets, selected text, and client rects. Image annotations use the same surrounding DOM context plus normalized image coordinates.
  20. Anchor state mapping: comment anchor state values are mapped, stale, and unmapped. A comment is mapped when the current version resolves the same data-plan-node-id or generated selector and the text quote still matches. It is stale when only a weaker selector or heading/text context match succeeds. It is unmapped when no current node/image can be resolved; the UI must still show the original captured context and screenshot.
  21. Stable node ID algorithm: preserve an existing id as data-plan-node-id; otherwise derive headingPathSlug + shortHash(textContent) for headings/sections; otherwise derive tagName + domSiblingPath + shortHash(trimmedTextOrOuterHtml). When two generated IDs collide within one version, append deterministic sibling-order suffixes -2, -3, etc. The renderer must produce deterministic IDs for identical plan content across registrations.
  22. Marker screenshots: the parent review shell captures a cropped marker screenshot with html2canvas after the selected-region highlight and marker are rendered. No screenshot code runs inside the sandboxed iframe. The screenshot is posted with the comment as safe PNG bytes, stored as a comment asset, and referenced by screenshotAssetId.
  23. Reviewer identity: the no-auth UI asks for an optional display name on first comment and stores it in localStorage. If omitted, comments use { "type": "reviewer", "displayName": "Anonymous reviewer" }. The API accepts optional createdBy.displayName and normalizes missing/blank values to that anonymous default.
  24. Agent delivery: comments are delivered at least once. Agents must claim then ack comments; unacked claims expire after a 5-minute lease and re-enter the pending queue.
  25. Main conversation response: comment events include a browser.comment.v1 conversation bridge payload so a host adapter can append the browser comment to the active chat thread, respond there, and ack/resolve the comment with a response summary, run ID, changed files, and optional commit SHA. The service/CLI owns the portable event and file/stdout handoff contract; host-specific injection into Codex, Claude, or Pi is a thin adapter that consumes that contract. The review service stores and displays response metadata but does not implement its own chat system.
  26. Queue state machine: comment status values are pending, claimed, acknowledged, and resolved. Allowed transitions are pending → claimed → acknowledged → resolved, pending → resolved, and claimed → pending on lease expiry or explicit release. Agent ack requires an active matching claim; direct ack from pending returns 409 claim_required. Ack and resolve are idempotent for the life of the comment.
  27. Event sequencing: event sequence numbers are monotonic per plan. SSE uses id: <sequence>; reconnect with Last-Event-ID replays later events. Retain events until both thresholds are exceeded: older than 7 days and outside the latest 10,000 events per plan.
  28. Latency/reconnect defaults: local SSE delivery target is p95 under 1 second after transaction commit. Heartbeats are sent every 15 seconds. CLI reconnect backoff starts at 1 second and doubles to a 30-second max, with REST polling every 10 seconds if SSE is unavailable.
  29. Concurrent claims: claiming is atomic. Two agents cannot hold the same active claim; conflicts return 409 claim_conflict with the current holder and lease expiry.
  30. Versioning: every registration creates or updates a plan version identified by file hash + repo + path + branch + commit. Comments attach to a plan version but can be shown on newer versions as “stale/mapped/unmapped.”
  31. Security posture: MVP intentionally optimizes for simple local-network access over access control. Authentication, reviewer tokens, private share links, SSO, and authorization are explicit future work, not part of initial execution.

Deployment and discovery

The MVP runs as one shared Homebrew/launchctl-managed daemon, not one service per repo. It binds to 0.0.0.0:4317 by default so other devices on the reachable network can open the plan index. Agents in any repo use the same CLI, which discovers the daemon URL from flag, environment, project config, user config, then the local default. The daemon stores every plan under a repo-scoped key so thoughts/plans/foo.html in two repos cannot collide.

Conversation bridge contract

The MVP should expose the comment event in a shape that the host agent can surface as a normal user message in the active conversation. The service does not need to own chat UI. It must provide enough metadata for Codex-style clients to render “Browser comment: Comment 1” with the selected text, marker screenshot, URL, selector, and reviewer note, then let the agent reply in the main thread. The ack/resolve payload records the agent response summary so the review UI can show that the comment was handled even though the full discussion happened in chat.

Agent watch contract

Agents keep an open watch connection with Server-Sent Events. WebSockets are intentionally not part of the MVP because comment creation, ack, resolve, and claim writes are ordinary REST requests, while the watch side only needs server-to-agent notifications. Add WebSockets later only if bidirectional live collaboration becomes a product requirement.

GET /api/plans/plan_123/events?mode=queue
Accept: text/event-stream
Last-Event-ID: 42
  • Primary transport: plan-review watch plan_123 --mode queue opens GET /api/plans/:planId/events?mode=queue with Accept: text/event-stream and keeps the HTTP connection open until the process exits, the server closes it, or the network fails.
  • Event shape: each SSE message uses id: <sequence>, event: <eventType>, and JSON data. Queue mode emits comment.created, comment.claimed, comment.acknowledged, comment.resolved, and comment.released. mode=all also emits plan-version events.
  • Resume: the CLI stores the latest delivered sequence per plan. On reconnect it sends Last-Event-ID; the server replays events with a greater sequence before streaming live events.
  • Heartbeat: the server sends event: heartbeat every 15 seconds with { latestSequence, serverTime } so agents can detect dead connections.
  • Fallback polling: if the SSE connection cannot be established or repeatedly drops, the CLI polls GET /api/plans/:planId/events/poll?afterSequence=&mode=queue every 10 seconds. Poll responses use the same event objects and monotonic sequence IDs as SSE, with retryAfterMs available for backoff.
  • Delivery semantics: watch delivery is at-least-once. Agents must dedupe by { planId, sequence }, then claim/ack comments through the queue endpoints before acting.
  • CLI output: --json prints each event as one NDJSON object with source: "sse" or source: "poll". --format browser-comment prints only browser.comment.v1 envelopes for host conversation adapters.

Claim request precedence

  • mode="one" requires no commentIds; it claims the oldest pending comment by sequence, respecting optional limit=1.
  • mode="selected" requires non-empty commentIds; limit is invalid and returns 400 validation_failed.
  • mode="bulk" rejects commentIds; it claims up to limit pending comments, default 50, max 200, ordered by sequence.
  • Expired claims are released transactionally before each claim query. Active claims held by another agent are skipped for bulk and returned as 409 claim_conflict for selected mode.

SSE event contract

id: 43
event: comment.created
data: {"sequence":43,"planId":"plan_01...","commentId":"cmt_01...","comment":{...}}
  • comment.created: new pending comment available, including conversationPayload.type = "browser.comment.v1" so the host adapter can surface the browser comment in the active conversation.
  • comment.claimed: agent claimed comment; review UI can show “agent reviewing.”
  • comment.acknowledged: agent consumed comment and may include agentResponse metadata for the review UI sidebar.
  • comment.resolved: comment closed, with resolution note and optional agent response summary.
  • comment.released: claim released or lease expired.
  • plan.version.registered: agent uploaded a new plan version.
  • heartbeat: every 15 seconds with { latestSequence, serverTime }.

Conversation bridge host contract

The daemon and CLI expose a host-neutral bridge; they do not directly control proprietary chat UIs. plan-review watch --format browser-comment prints one browser.comment.v1 envelope per pending comment. plan-review watch --conversation-out ~/.plan-reviewer/host-bridge/inbox.ndjson appends the same envelopes to a durable file for Codex/Claude/Pi host adapters. Host adapters are responsible for inserting that envelope into the active conversation. The MVP includes a serializer/parser and a fixture adapter test; if a host lacks an active-thread injection API, the adapter must fail with an actionable message and leave the comment pending.

CLI output contracts

  • plan-review register path.html --json prints the register response JSON and exits 0. Human mode prints Plan ID, Index URL, Review URL, and Watch command. Registration uploads or references relative image assets found in the plan.
  • plan-review index --json prints the GET /api/plans response JSON. Human mode prints the index URL and a compact table of registered plans.
  • plan-review watch plan_123 --json opens the SSE watch endpoint and prints one newline-delimited JSON event per line, reconnecting with saved sequence state. --mode queue prints only queue-relevant comment events, --format browser-comment prints the conversationPayload envelope, --conversation-out <path> appends those envelopes to a durable NDJSON host-bridge inbox, --once exits after the first matching event or timeout, and --timeout <ms> exits non-zero when no event arrives in time. If SSE is unavailable, 10-second REST polling against /api/plans/:planId/events/poll emits the same event objects with "source":"poll"; SSE events use "source":"sse".
  • plan-review queue claim plan_123 --all --json maps to POST /api/plans/:planId/comments/claim with { "mode": "bulk" }, prints the claim response JSON, and exits 0 when zero or more comments are claimed. --one maps to { "mode": "one" }; --ids cmt_1,cmt_2 maps to { "mode": "selected", "commentIds": [...] }.
  • plan-review ack cmt_123 --claim claim_123 --json and plan-review resolve cmt_123 --json print their endpoint response JSON and are idempotent. Agent ack without --claim exits non-zero and prints the claim_required next action.

  1. Homebrew installs the daemon/CLI from this repo checkout as a local tap with brew tap local/ai-configs <repo-checkout-path> and brew install local/ai-configs/plan-reviewer, and brew services start plan-reviewer runs it persistently through launchctl on 0.0.0.0:4317 by default.
  2. Agents can register an HTML plan from thoughts/plans via CLI and REST, receiving a stable plan ID, index URL, and review URL.
  3. The index at GET / lists registered plans across repos with search/filtering, latest metadata, status counts, and links to open/comment on each plan.
  4. The review URL renders the plan in a dark-compatible review shell without executing arbitrary plan-authored JavaScript.
  5. Reviewers can select a DOM element/range or image/image region in the rendered plan, write a comment, and see an anchored marker, selected-region highlight, comment composer, and sidebar entry matching the UX in Review UX and agent response loop.
  6. Each comment captures enough context for an agent to locate the affected plan content without looking at the browser: selectors, DOM path, heading path, quote, coordinates, marker screenshot, trimmed HTML/text context, and image coordinates/assets for image comments.
  7. Images render from safe copied assets or safe data/blob URLs, support zoom/pan in a popover/lightbox, and support whole-image, point, and rectangle comments.
  8. Agents can monitor comments near-real-time through a documented open SSE watch connection, receive a browser.comment.v1 payload through stdout or a host-bridge NDJSON inbox for active chat insertion, and recover missed events through sequence replay or the documented REST polling fallback. WebSockets are not required for MVP.
  9. Multiple comments queue durably. Agents can claim one comment, selected comments, or all pending comments for a single plan; repo-wide processing uses GET /api/agent/queue?repoKey=... followed by per-plan grouped claims.
  10. Unacked claimed comments return to pending after a lease timeout; duplicate delivery is safe because ack is idempotent.
  11. After claiming the comment and responding in the main conversation through a host adapter, an agent can ack or resolve the comment with response summary, changed files, run metadata, and optional commit SHA; the review UI shows that metadata on the anchored marker/sidebar entry.
  12. CLI and REST expose equivalent registration, queue, claim, ack, resolve, and watch capabilities.
  13. Errors for ambiguous repo inference, invalid HTML, missing image assets, unavailable service, and stale anchors include exact next steps.
  14. Docs explain how agents should generate HTML plans with stable IDs and how to register/watch them.

Given an agent is running plan-review watch plan_123 --mode queue
When a reviewer creates a comment
Then the CLI receives a comment.created event within the p95 under-1-second local stream latency target and prints structured JSON with conversationPayload.type = "browser.comment.v1" suitable for a host adapter to append the browser comment to the active agent conversation.

B4 - Bulk processing

Given five pending comments exist for a plan
When the agent runs plan-review queue claim plan_123 --all
Then the service marks all five as claimed by that agent lease and returns them in deterministic sequence order.

B5 - Disconnect recovery

Given an agent stream disconnects after event sequence 10
When it reconnects with Last-Event-ID: 10 or CLI state containing sequence 10
Then the service replays later events and the agent does not lose comments.

B6 - Lease expiry

Given an agent claims a comment but never acks it
When the 5-minute claim lease expires
Then the comment returns to pending and a future queue claim can receive it again.

Then the index groups plans by repo, shows latest version and comment counts, supports filtering, and links to each review page.

B11 - Agent response reflected in review UI

When the agent updates the plan and calls plan-review ack cmt_123 --claim claim_123 --note "handled in plan update" --json with response metadata

Delivery order

  1. Package skeleton, Homebrew formula, and launchctl service.
  2. Registration, storage, plan index, image asset handling, and safe rendering.
  3. Reviewer UI with DOM and image anchoring, including selection visuals, composer, marker, and sidebar states.

  5. Queue claim/ack/resolve lifecycle with agent response metadata shown in the review UI.
  6. Docs, dogfood workflow, and future security seams.

Non-goals

  • General-purpose website QA or bug reporting outside registered HTML plan files.
  • Realtime collaborative editing of the plan document.
  • A standalone chat product inside the review service; agent replies happen in the host conversation and are summarized back into the review UI.
  • Replacing existing markdown inline plan review commands in Phase 1.
  • Internet-hosted public SaaS plan hosting; the MVP is a local-network daemon and intentionally has no auth.
  • Bearer tokens, private share links, SSO, OAuth, or enterprise administration in the initial release.
  • Guaranteeing exactly-once event delivery; the queue contract is at-least-once with idempotent ack.
  • Perfect automatic remapping of anchors across arbitrary plan rewrites.

Risks and mitigations

RiskMitigation
Unsafe plan HTML executes arbitrary code.Sandbox iframe, CSP, block plan-authored scripts by default, inject only service overlay.
DOM anchors break after plan regeneration.Capture multiple selectors and original context; show stale/unmapped status instead of pretending confidence.
Agents miss comments while offline.Durable event log, sequence replay, REST queue snapshots, claim leases.
No-auth service is exposed on the network.Keep this intentional for MVP, bind to 0.0.0.0 by default per product requirement, print a startup warning, and document host override for local-only use.
Reviewers accidentally comment on decorative markup.Overlay should bubble to nearest semantic element and display selected context before submit.
Browser comments are disconnected from the active agent conversation.Every comment event includes a conversation bridge payload; ack/resolve stores response metadata that the review UI can display even though the full reply lives in chat.
Image anchors drift after resize or reload.Store natural dimensions plus normalized point/rectangle coordinates and recompute display positions from current rendered dimensions.
CLI and REST drift.Generate CLI client behavior from shared schemas/contracts and test both surfaces against the same fixtures.

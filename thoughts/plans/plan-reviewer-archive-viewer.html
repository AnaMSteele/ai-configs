Goal

Make archived plans discoverable from the plan viewer so a user can switch from active plans to archived plans, inspect what was archived, and recover from accidental archive without relying on re-registering a plan as hidden tribal knowledge.

Why this plan exists

The archive backend already existed enough to hide archived plans from the index, but the user-facing workflow was incomplete: once a plan was archived, the normal index no longer exposed an archive view or restore path. This plan defined the UI direction before implementation and now records the implemented outcome.

Authority and inputs

  • User request: create an HTML plan, include a few UI variations, use Claude Code to design the UI, present it through the viewer, and do not change product code.
  • Repo guidance: AGENTS.md says this repo works directly on main, reviewed HTML plans live under thoughts/plans/, and product code should not change during planning.
  • Skill guidance: planning-workflow, html-plan-reviewer, reviewed-html-plan, product-principles, and claude-code-review.
  • Claude Code read-only design pass: compared three directions; user selected the dedicated archive page direction and removed the alternatives.

Current implementation reality

  • indexHtml() renders the plan review index as a server-generated dark page with search, repo filter, progress bars, comment counts, and an Archive button per card.
  • GET / currently calls store.listPlans(), which excludes archived plans by default.
  • GET /api/plans?includeArchived=true already includes archived plans and exposes plan.archivedAt.
  • POST /api/plans/:planId/archive exists and sets archived_at.
  • There is no explicit unarchive/restore endpoint. Re-registering a plan currently clears archived_at, but that is not an obvious recovery workflow.
  • The review shell can open an archived plan by ID, but its nav button still presents Archive plan rather than restore-aware state.

Product intent alignment

The right action should be obvious and recoverable. Archiving should remove noise from the active index, but not make plans feel lost. The plan viewer should show archived state where the user naturally looks, provide a safe restore path, and avoid exporting recovery to a separate re-registration command.

Locked decisions

  • Archived plans remain hidden from the default active index and are viewed from a dedicated /archive page.
  • The dedicated archive page must reuse existing plan card metadata: repo, slug, path, progress, comment counts, activity, and archivedAt.
  • Recovery must become explicit as Restore/Unarchive, not implicit through re-registration.
  • The archive UI should keep the current dark-mode plan-reviewer visual language and provide a clear return path to the active index.
  • Plan-authored mocks in this file are static. The reviewer shell owns interactivity.
  • Re-registering or filesystem source sync must preserve archive state; only the explicit Restore/Unarchive action should move a plan back to active.
  • Archive and restore should be safe for stale tabs: repeat Archive or Restore calls should be idempotent enough to leave the plan in the requested final state and return truthful metadata.

Selected direction

Build the dedicated archive page.

The selected design is the former Option C: a dedicated /archive page reached from a header-level Archived (count) → link on the normal active index. The active index remains the default golden path, while archived plans get a full browser page with search, repo filter, sort, Open, and Restore actions.

Acceptance criteria

  1. The default index continues to show only active plans in the same card layout users have today.
  2. The user can reveal archived plans from the index without knowing an API flag or plan ID.
  3. Archived plans show enough context to identify them: repo, slug, path, archived timestamp, progress, and comment counts.
  4. The user can restore an archived plan explicitly from the archive viewer.
  5. After restore, the plan appears in the active index and no longer appears in the archived view.
  6. Empty archived state is clear and non-alarming.
  7. Opening an archived plan directly does not show a misleading archive-only primary action.
  8. Re-registering or filesystem source sync for an archived plan does not silently restore it to the active index.
  9. Restore failures keep the archived card visible and provide retry guidance.

BDD scenarios

Archived plan discovery

Given at least one plan has archivedAt, when the user opens the plan index and follows the Archived (count) → link, then /archive shows the archived plan with repo, slug, path, archive time, progress, and comment counts.

Active index remains default

Given active and archived plans exist, when the user opens /, then active plans remain primary and archived plans are not mixed into active card groups; archived plans are reached through the index-level archive link.

Restore from archive

Given a plan is archived, when the user clicks Restore on /archive, then the system clears archivedAt, removes the card from archived results, and the plan is available in active results.

Empty archive

Given there are no archived plans, when the user opens the archive page, then the page shows a quiet empty state: “No archived plans yet.”

Archive count accuracy and staleness

Given the index header shows Archived (N) →, when a plan is archived, restored, or re-registered, then the count reflects the current archived set after reload and never counts active plans.

Active/archive partition is strict

Given one active and one archived plan share similar slugs or paths, when the user searches either page, then active results appear only on / and archived results appear only on /archive.

Filtered archive has no matches

Given archived plans exist, when the user applies a repo or text filter that matches none of them, then /archive shows a scoped empty state explaining that no archived plans match the current filters and offers a clear way to clear filters.

Archive sort order is deterministic

Given multiple archived plans have different archive and activity times, when the user opens /archive, then the default sort is newest archived first and ties are stable so cards do not jump unpredictably across reloads.

Restore failure is recoverable

Given a restore request fails because the plan no longer exists, the service is unavailable, or validation fails, when the user clicks Restore, then the card is not silently removed and the UI shows an actionable error with retry guidance.

Double restore and stale tabs are safe

Given two browser tabs show the same archived plan, when the plan is restored in one tab and the user clicks Restore in the stale tab, then the operation is idempotent or returns a clear already-restored state without re-archiving, duplicating, or erroring cryptically.

Open archived plan preserves context

Given an archived plan has comments and progress, when the user clicks Open from /archive, then the existing review shell opens that plan and clearly indicates the plan is archived with a path back to /archive or the active index.

Review shell handles archived state

Given the user opens an archived plan directly at /p/:planId, when the shell loads metadata, then it does not present a misleading “Archive plan” primary action; it should show Restore or archived status instead.

Re-registering an archived plan does not surprise the user

Given current storage clears archivedAt on register, when an archived plan is re-registered after this work, then archivedAt is preserved and only the explicit Restore/Unarchive action moves the plan back to active.

Archived source-sync plans stay quiet

Given an archived plan has filesystem source sync enabled, when the source file changes or the watcher scans filesystem plans, then source sync does not silently restore the plan to active; continuous syncing of archived plans remains out of scope.

Comments on archived plans remain reachable

Given an archived plan has pending or resolved comments, when the user opens it from /archive, then comment counts, anchors, and queue status remain visible and no comment data is lost or hidden by the archive state.

Keyboard and accessible navigation

Given a keyboard-only user opens the index or archive page, when they tab through controls, then Archived, Active index, Open, Restore, filters, and sort controls are reachable in a logical order with descriptive labels.

Small viewport layout remains usable

Given the archive page is viewed on a narrow browser window, when filters and archived cards wrap, then critical actions remain visible, cards do not overflow horizontally, and the Active index return path remains near the top.

Direct archive route and refresh work

Given the user bookmarks or refreshes /archive, when the route loads without prior index state, then it fetches/renders archived plans directly and does not depend on client state from /.

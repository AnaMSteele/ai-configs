HTML Plan Reviewer Draft-Safe Refresh UX

Prevent live plan sync from destroying an in-progress browser comment, and close the loop on why recently added review-shell UX is not visible in the running browser.

Status

Execution-ready. The desired behavior is known, the affected files are identified, and verification can be driven through the existing Playwright e2e fixture.

Goal

When a registered plan changes in the background, the review shell must not reload the rendered iframe or clear comment state while a reviewer has an open comment composer. The reviewer can finish or cancel the comment, then the latest plan version applies safely.

The same implementation pass must also make the served browser surface visibly current: Cmd+Enter/Ctrl+Enter submits comments, and the top navigation bar with the Plan Index link is present on the running service.

Why this plan exists

Authoritative source sync currently improves plan freshness at the cost of reviewer safety. If the repo file changes while someone is typing, the review page calls its reload path, clears the pending selection, empties the textarea, and hides the composer. That makes background agent edits hostile to the reviewer’s active workflow.

The user also reports that previously requested UI changes are not visible in their running browser. Source and the local service indicate those changes exist, so the plan includes a delivery/served-artifact verification phase rather than assuming code changes alone are enough.

Current implementation reality

  • loadMeta({ reloadPlan: true }) in app.ts calls clearPendingSelection() before changing frame.src. That explicitly clears selected, pendingAnchor, comment-body, the composer, and the lightbox.
  • handlePlanVersionEvent() calls loadMeta({ reloadPlan: true, forceReloadPlan: event.type === 'plan.version.synced' }), so filesystem sync can trigger the destructive path even when the latest version ID is reused for asset-only changes.
  • The current source has a comment-body keydown handler that submits on Enter with Meta or Ctrl and preserves Shift+Enter as a newline.
  • The current source and locally running http://127.0.0.1:4317/p/<planId> shell include #plan-navbar with a “← Plan index” link and Archive button.
  • The e2e fixture proves Cmd/Ctrl+Enter and basic plan sync, but it does not cover “sync arrives while composer is open with typed text.” That is the missing regression guard.
  • Because the user’s browser does not show UI that the running local endpoint now serves, delivery verification must check the exact user-facing URL, browser cache/module behavior, and Homebrew-installed service artifact.

Product intent alignment

Default workflow contract

Agent edits should live-sync plan content, but never interrupt the reviewer’s active comment. The correct path is automatic: the reviewer should not need to copy text elsewhere, manually pause source sync, or reload at the right moment.

Safe defaults and recovery

While a comment composer is open, the review shell keeps showing the old rendered iframe and stores exactly one pending latest-version refresh. Comment metadata remains tied to the version and anchor the reviewer actually selected. After submit or cancel, the shell applies the latest deferred version and remaps comments using existing anchor-state rules.

Fail-closed boundary

If the selected anchor becomes invalid only after the deferred refresh, the submitted comment should still be accepted against the version where it was created; any later mapping can become stale or unmapped. Do not silently mutate the user’s selection to a new node while they are typing.

Locked decisions

  1. A comment dialog is considered open when the composer is visible and either pendingAnchor exists or the textarea has draft text.
  2. Plan version events received during an open composer are deferred, not ignored. Only the latest deferred version needs to be applied.
  3. Comment sidebar data and sync warnings may still update while a draft is open, but iframe reload, selection clearing, textarea clearing, and lightbox hiding must not happen until the draft resolves.
  4. “Dialog resolved” means successful submit or explicit Cancel. Successful submit clears the draft and then applies the deferred refresh. Cancel discards the draft and then applies the deferred refresh.
  5. When a refresh is deferred, show a small, non-blocking notice near the composer/sidebar: “Plan updated in the background. Finish or cancel this comment to refresh.”
  6. Do not reload the whole page for source sync. The only refresh target is the rendered iframe, and even that is deferred while the composer is open.
  7. Serve the review shell, /client.js, and /client.css with explicit no-cache or revisioned URLs so local browser/module cache cannot hide newly shipped UI.

Acceptance criteria

  1. If a source sync event arrives while the composer is open and contains typed text, the textarea value, pending anchor, selection UI, and submit ability remain intact.
  2. After the reviewer submits the comment, the comment is created successfully and the latest deferred plan version becomes visible without a full page reload.
  3. After the reviewer cancels the comment, the latest deferred plan version becomes visible and no empty comment is created.
  4. Existing comments, claim/ack/resolve changes, and sync failure warnings can still update while a draft is open without clearing the draft.
  5. Cmd+Enter on macOS and Ctrl+Enter elsewhere submit the active composer in the running browser; Shift+Enter still inserts a newline.
  6. The running review page has a top navigation bar with a working link back to the plan index.
  7. The implementation verifies the actual served URL the user opens, not only TypeScript source or freshly built dist files.

End State

Tests first

Run the P1 e2e test and keep it failing until the state machine is implemented. Add a narrow unit-style contract only if helper extraction makes that cheaper than browser coverage.

Work

  • Add small client helpers inside clientJs: hasOpenCommentDialog(), queueDeferredPlanRefresh(), and applyDeferredPlanRefreshIfIdle().
  • Split metadata refresh from iframe reload. loadMeta() should be able to update comments/sync warning without clearing selection or changing frame.src.
  • Change handlePlanVersionEvent() so it queues reload details when hasOpenCommentDialog() is true, otherwise it reloads the iframe as today.
  • After successful submitPendingComment(), clear the submitted draft, call loadMeta(), then apply the deferred latest refresh if present.
  • On Cancel, clear the draft and then apply the deferred latest refresh if present.
  • Add a small #deferred-refresh-notice element in the shell/sidebar/composer area and hide/show it from the queued-refresh state.

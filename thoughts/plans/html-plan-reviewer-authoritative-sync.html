HTML Plan Reviewer Authoritative Source Sync

Make the repository plan file the single source of truth. The service may keep rendered/cached copies, but those copies must be derived from and refreshed from the authoritative file after registration.

Goal

Change tools/plan-reviewer so registering thoughts/plans/<plan>.html creates a durable link to that repo file, not a one-time detached snapshot. After registration, edits to the file should update the service-rendered plan automatically and notify open review pages.

The golden path should become:

plan-review register thoughts/plans/my-plan.html --repo auto --branch auto --commit auto
# edit thoughts/plans/my-plan.html
# browser review page updates to the latest registered/synced version

Non-goals

  • Do not build an in-browser editor for repo files in this phase.
  • Do not support remote repos where the service cannot access the local filesystem.
  • Do not add authentication or sharing controls; preserve the current trusted-network MVP posture.
  • Do not delete existing stored versions or comments; migrate/add fields compatibly.

Current behavior

  • The CLI reads the plan file once in src/cli.ts registerPlan and sends html, fileHash, and copied relative image assets to POST /api/plans/register.
  • The service stores HTML/rendered blobs under ~/.plan-reviewer/blobs and serves /render/:planId from the latest stored rendered blob.
  • No service component watches the repo file or re-reads it after registration.
  • The browser shell listens for comment events only, so even an explicit re-register requires a page reload to show the new HTML.

Recommended architecture

Decision: central service owns filesystem watching

Implement direct source watching inside the existing Homebrew service. The service already runs as the logged-in user LaunchAgent, is not sandboxed, and can normally read /Users/anichols/code/**. Do not add a second daemon, RPC protocol, lifecycle manager, port allocation, or helper cleanup path.

Keep source access behind a narrow SourceProvider interface for testability and future internal refactors, but the only planned provider for this work is FilesystemSourceProvider inside the central service.

1. Register
CLI sends the authoritative absolute source path plus the initial HTML/assets snapshot.
2. Watch
Service records the source path, starts a debounced watcher, and reuses the registration pipeline on file changes.
3. Refresh
Service writes a new derived version, emits a plan-version event, and open browsers reload the iframe.

Source-of-truth rule

The repo file is authoritative. Service blobs are a cache and historical version store. Any sync failure should leave the last good rendered version visible with a clear warning/event, not silently diverge.

P1: Contracts

  • Update src/schemas.ts so registration accepts optional source metadata while preserving existing snapshot registrations.
  • Add explicit event names for version sync and sync failure. Keep mode=queue comment-only; use default/all mode for browser plan-version updates.
  • Add contract tests that prove a re-register of the same source path keeps the same planId and creates a new latest version only when the file hash changes.

P2: Storage and source metadata

  • Extend PlanReviewStore schema creation/migration in src/storage/database.ts with source/watch fields. Existing DBs must continue to open.
  • Return source sync status from registration so CLI JSON and browser metadata can show whether the plan is live-linked or snapshot-only.
  • Expose source path and last sync status in GET /api/plans and GET /api/plans/:planId.

P3: Filesystem watcher

  • Add chokidar as a runtime dependency instead of relying on raw fs.watch; this is a maintained library and avoids hand-rolled cross-platform watch edge cases.
  • Create src/server/sourceSync.ts with a narrow service that can register(planId), unregister(planId), rehydrateFromStore(), and syncNow(planId, reason).
  • For filesystem mode, read source_path, discover relative image assets from the plan directory, call renderPlan, and store a new version through the same store path as manual registration.
  • Debounce file changes for 300-750ms, ignore unchanged file hashes, and serialize per-plan syncs to avoid races during editor save bursts.
  • On service startup, rehydrate all watch_mode="filesystem" plans and run an initial sync check. Missing files should set last_sync_status="failed" and emit/log an actionable error.
  • If the service cannot read the path, keep serving the last good cached version and report: path, errno, suggested fix, and fallback command plan-review register <path> --snapshot if snapshot mode is retained.

P4: Browser auto-refresh

  • Have the review shell listen for plan.version.registered, plan.version.synced, and plan.sync.failed in its existing EventSource.
  • When latest version changes, call loadMeta(), update versionId, and reload the iframe with a version query string like /render/:planId?versionId=<id> to avoid stale browser cache.
  • Preserve comment sidebar state and markers after reload. Existing comments may map/stale/unmap according to the current anchor mapping rules.
  • Display a visible sync warning in the sidebar when the latest sync failed, while leaving the last good rendered plan visible.

P5: CLI, docs, and skill updates

  • Make plan-review register default to live filesystem sync when it can resolve an absolute source path. Print “Source sync: active” with the watched path.
  • Add explicit flags only if needed: --snapshot for detached historical review and --sync-now for manual repair. Do not add helper daemon commands.
  • Update tools/plan-reviewer/README.md to state that repo files are authoritative and service blobs are derived cache/history.
  • Update /Users/anichols/.agents/skills/html-plan-reviewer/SKILL.md so agents know they no longer need to re-register after every edit once source sync is active.

No helper service

Do not plan or implement a repo-local helper daemon

This plan assumes the central Homebrew service can watch and read registered plan files directly. If a source file cannot be read, the required behavior is an actionable sync failure plus continued serving of the last good rendered version, not a helper-service fallback.

Only revisit a helper/RPC design if direct central-service filesystem watching is proven impossible by verification evidence and the user explicitly reopens that scope.

Risks and mitigations

  • Editor save bursts: debounce and serialize syncs per plan; ignore unchanged hashes.
  • macOS permissions: service is a user LaunchAgent, so normal repo paths should work. Protected folders may fail; expose sync errors and keep last good render.
  • Stale anchors: comments remain tied to their version and are mapped to the latest render using current anchor-state logic.
  • Cache confusion: label stored blobs as derived versions in docs/UI and always show source path plus latest sync status.

Acceptance criteria

  1. The registered repo file is documented and represented in API responses as the authoritative source.
  2. Changing that file updates the latest service-rendered version without manually re-running plan-review register.
  3. An open review page updates to the latest version after a source sync event.
  4. If the source file cannot be read or rendered, the service keeps serving the last good version and surfaces the failure.
  5. Homebrew service restart preserves registered live watches and resumes syncing.
  6. README and html-plan-reviewer skill describe the new authoritative-source workflow.

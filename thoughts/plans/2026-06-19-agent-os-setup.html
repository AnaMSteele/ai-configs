Agent Operating System Setup Plan

Status: execution-ready for local tooling and repo-doc setup only. No production data, infrastructure, secrets, deploys, or product code are in scope.

Goal

Make Ana's ai-configs repo the source of truth for local agent skills, hooks, and plan-review tooling used in future work in heddle, doct, and related repos, while preserving repo-local AGENTS.md / CLAUDE.md authority.

How to Invoke the Planner

For new non-trivial work, ask the active agent to use the reviewed-plan workflow before implementation. Pi users can invoke /dev:reviewed-html-plan <task> for a planning-only browser review gate, or /skill:adn-dev-wf <task|plan> for the full reviewed-plan workflow that can continue into execution after comments are settled.

Claude Code, Codex, Hermes, and other non-Pi agents do not need Pi slash commands. They should load the same repo-owned skills/reviewed-html-plan and skills/html-plan-reviewer payload through their installed skill, prompt, command, or orchestration adapter; create or refresh a semantic HTML plan under thoughts/plans/; register it with plan-review; return the browser URL to the user; and monitor the queue-backed comment flow before execution proceeds.

Non-Claude Agent Flow

Claude-specific skill paths are only one consumer surface. Codex, Pi, Hermes, and other non-Claude agents should load the same repo-owned planner payload through their supported prompt, skill, command, or adapter mechanism rather than copying or re-deriving the workflow.

When a non-Claude agent handles plan-review comments, it should use http://127.0.0.1:4317 for local plan-review CLI calls, use http://mbp.braid-python.ts.net:4317 for human-facing review links, claim and process exactly one comment at a time, and acknowledge or resolve through plan-review ack / plan-review resolve metadata.

BDD Scenarios

Plan publication

Given the local service is running, when this file is registered with plan-review register, then the reviewer returns a browser URL and source sync remains active.

Skill load

Given Claude currently looks in ~/.claude/skills and other agents such as Codex or Hermes need their own supported skill, prompt, or command surfaces, when a future agent requests the standard workflow, then it loads the same repo-owned canonical payload through a consumer-specific symlink or adapter instead of copying or re-deriving behavior.

Origin alignment

Given local ai-configs changes exist, when an agent finishes work, then those changes are not done until they are committed and pushed to origin at AnaMSteele/ai-configs.

Central path blocked

Given ~/.agents is root-owned, when the canonical installer tries to create ~/.agents/skills, then the setup reports the blocker and uses Claude symlinks as a reversible local fallback.

Codex comment response

Given a reviewer leaves a pending browser comment, when plan-review-codex-responder.sh runs, then it claims one comment, gives the claim JSON and repo context to Codex, and expects Codex to update the plan or acknowledge the comment using the active claim.

User planning flow

Given a user wants an agent to use the planning tool for a new task, when they either invoke Pi's /dev:reviewed-html-plan <task> or /skill:adn-dev-wf <task|plan>, ask Claude Code or Codex to use the reviewed HTML plan workflow, or have Hermes route to the same repo-owned reviewed-plan adapter, then the agent creates or resumes the HTML plan, registers it with plan-review, opens the browser-review URL for feedback, and uses the queue-backed comment flow before execution.

Ana Agent Ops ccore guidance

Given Ana has an Agent Ops ccore space for shared agent guidance, runbooks, and skills, when an agent needs cross-agent operating instructions, then it consults the documented ccore access path and usage rules while preserving repo-local AGENTS.md / CLAUDE.md authority for project-specific constraints.

Responder failure

Given Codex fails while handling a claimed comment, when the responder exits that attempt, then it releases the claim so the comment is not stranded.

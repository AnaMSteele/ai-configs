Claude Review Reliability Plan

Status

Implementation complete on branch claude-review-reliability. All P0-P4 checkboxes are complete, source and installed guardrails/smokes pass, and Codex implementation review v5 returned VERDICT: PASS_SCOPED. The required Claude implementation review was attempted through the canonical launcher and failed closed with CLAUDE_SESSION_LIMIT_IN_TUI because the real Claude account session/rate limit is currently active; no alternate Claude transport was used.

Goal

Make every required Claude Code review in this repository use one canonical interactive Claude launcher: a real claude TUI created inside a launcher-owned private tmux server and driven by paste/send/capture. No required review path may use claude -p, prompt piping, direct claude shell snippets, direct interactive_shell/process launches, or model-provider substitutes.

Why this plan exists

The last week of Pi sessions showed Claude review reliability failures and repeated rework when agents mixed direct claude -p, tmux-wrapped claude -p, and direct interactive launches. The user specifically asked for a reviewed plan before repo changes because claude -p uses a different token/session path than interactive Claude and has produced misleading login/time-out failures.

Current implementation reality

Claude review workflows currently mix three launch styles:

  1. direct claude -p from process or shell,
  2. claude -p inside tmux,
  3. real interactive Claude sessions.

The first two do not satisfy the requirement. Existing evidence:

  • 2026-06-06 ... heddle-bunyip: PR186 review attempted one tmux claude -p run, then at least eight direct claude -p retries. The tmux auth probe returned loggedIn: false, authMethod: none, and EXIT=1. Direct claude -p attempts were killed or exited without useful output; one artifact only contained Warning: no stdin data received in 3s....
  • 2026-06-01 ... heddle-dev-worktrees-auto-update: tmux plus claude -p worked in 2-3 minutes, showing the old path can work but still uses the wrong non-interactive token path.
  • 2026-06-05 ... plan-reviewer: tmux plus claude -p plan and rereview succeeded, proving reliability varies by session state.
  • 2026-06-05/06 ... NOD-906: four tmux plus claude -p plan review passes succeeded, but required multiple reruns for plan convergence.

Local auth checks during planning showed:

  • direct current shell: claude auth status returns loggedIn: true,
  • existing codex tmux session: claude auth status returns loggedIn: false,
  • private tmux socket created from the current shell: claude auth status returns loggedIn: true.

Implication: tmux has no independent auth state. Claude credential availability depends on the parent process/environment that creates the tmux server and runs the CLI.

Mock and smoke validation already performed without modifying repo files:

  • Direct claude -p path rejected by mock.
  • Interactive tmux prompt paste plus sentinel capture succeeded in mock.
  • Auth failure path returned an explicit preflight failure.

  • Real private-tmux smoke from this Pi process reached loggedIn: true, while existing codex tmux returned loggedIn: false.
  • An early interactive Claude review driver had a false-positive sentinel bug because it matched prompt echo; this plan now requires post-submit/post-clear boundary tests.

Progress

Resume instructions (agent)

Read this whole plan first. Identify the first unchecked item in Progress, execute only that phase, update its checkbox immediately after successful verification, then continue phase-by-phase. Do not modify repo files before this plan is approved. During execution, do not create alternate Claude launch transports, do not use claude -p for required reviews, and do not mark the work done until the final source and installed guardrail scans plus Pi/OpenCode same-process smokes pass. If a Claude auth or TUI readiness check fails, preserve the transcript and inspect command, stop with the classified error, and ask for the exact user action named by the failure; never retry by changing transport.

Product intent alignment

  • Golden path: agents call one canonical launcher; the launcher owns tmux, readiness, prompt delivery, answer extraction, failure classification, and transcript preservation.
  • Safe default: create a fresh private tmux server from the real review-launching process. Do not reuse the existing codex tmux session, because it currently reports loggedIn: false.
  • Self-healing boundary: the launcher can verify prerequisites, classify failures, preserve inspectable tmux state, and point to exact smoke commands. It must not silently switch to claude -p, --file, prompt piping, direct interactive_shell, or model-provider substitutes.
  • Fail-closed boundary: identity/auth, TUI readiness, and prompt-echo boundary uncertainty must stop execution. A false successful review is worse than a blocked review.
  • Agent-legible errors: missing tmux/claude, claude auth status false, TUI not logged in, TUI readiness timeout, prompt-boundary uncertainty, and review timeout all include what was tried, where the raw transcript is, how to inspect tmux, and the next safe action.

Locked decisions

  • Required Claude Code reviews use only skills/claude-code-review/scripts/claude_interactive_review.py or its installed copies.
  • The launcher starts exactly one real interactive claude TUI inside a private tmux server and drives it through tmux paste/send/capture.
  • Required reviews must never use claude -p, claude --print, direct prompt args, input redirection into claude, stdin/prompt piping into claude, --file, future unknown non-interactive Claude flags, direct interactive_shell/process/bash Claude snippets, raw tmux new-window ... claude outside the launcher, or model-provider substitutes.
  • Guardrail enforcement is allowlist-based for executable Claude Code CLI invocation candidates in review surfaces. Candidate invocations are defined by the guardrail detection grammar below; prose mentions, headings, directory/path names such as claude-code-review, filenames such as claude_interactive_review.py, slugs like claude-plan-*, environment/marker names such as CLAUDE_REVIEW_*, JSON metadata, and non-command explanatory text are not invocation candidates. Any candidate invocation outside the canonical launcher/test exemptions and outside the exact launcher-internal forms allowed only in canonical launcher copies fails, even if it does not match a known denylist pattern.
  • Anthropic/Claude model-provider reviews that are not Claude Code CLI launches, such as OpenCode Zen claude-opus-*, are optional non-gating reviewers only.
  • claude auth status in the launcher-created tmux server is an early signal only. Final readiness also requires live TUI prompt detection and post-submit answer-boundary extraction.
  • The existing codex tmux session is not the default review host.
  • The launcher validates transport completion only. Domain verdict parsing remains owned by callers such as _opencode/scripts/linear_build_review.py.
  • skills/install-matrix.json keeps class: "consumer-specific-installable" and sets claude-code-review.allowedConsumers to ['pi', 'opencode'].
  • OpenCode invokes the launcher via sys.executable, not by requiring executable bits.
  • Tests may fake Claude only by controlling temporary HOME/ZDOTDIR/PATH; production code has no fake-Claude or binary override flag except the explicit OpenCode resolver env CLAUDE_REVIEW_LAUNCHER for source validation.

Acceptance criteria

AC1. No Claude review skill, prompt, script, command, or documented required review process recommends or executes claude -p or another direct Claude transport.

AC2. All required Claude review paths call one deterministic launcher entrypoint; agents cannot choose alternate shell snippets, fallback transports, flags, or retry strategies.

AC3. The launcher creates a private tmux server from the real caller process, starts interactive claude, drives it through paste/send/capture, tears down successful review and smoke tmux servers, and preserves transcript/inspect metadata on failure.

AC4. Prompt-echo cannot satisfy completion. Success requires an answer-start marker and sentinel after a post-submit/post-clear boundary in Claude's answer region.

AC5. Mock tests pass without real Claude, including auth failures, timeout, prompt echo, ANSI/chrome, wrapping, and parseable output cases.

AC6. Same-process real smoke tests pass for every consumer that invokes the launcher: Pi prompt path and OpenCode linear_build_review.py source and installed paths.

AC7. Auth failure, missing prerequisites, TUI not ready, prompt-boundary uncertainty, and timeout produce classified, agent-legible failures with preserved transcript and inspect command.

AC8. Existing workflows still support plan review, implementation review, rereview, verdict extraction, and parent-agent artifact saving.

AC9. OpenCode linear_build_review.py continues to own verdict parsing while delegating only Claude transport to the launcher.

AC10. Source and installed guardrail scans pass across repo, ~/.agents/skills, ~/.config/opencode/skills, ~/.config/opencode/scripts, ~/.config/opencode/commands, ~/.config/opencode/agents, ~/.config/opencode/prompts, ~/.pi/agent/prompts, and ~/.pi/agent/agents; scans fail by default for any non-exempt executable Claude Code CLI invocation candidate in review surfaces that is not inside a canonical launcher copy or approved test fixture/marked forbidden example.

AC11. A real-Claude prompt-submitting end-to-end validation passes before completion: it submits a tiny live review prompt through the launcher, observes the real TUI readiness and post-submit/post-clear boundary, extracts only the answer-region marker/sentinel pair, and records the confirmed readiness regex plus clear-boundary observable used by production code.

BDD scenarios

Scenario S1 — Required review uses canonical interactive tmux launcher

  • Given an agent needs a required Claude Code review,
  • When it invokes the review workflow from Pi or OpenCode,
  • Then the workflow calls the canonical launcher,
  • And only the launcher starts cd <repo> && zsh -ilc 'claude' inside a launcher-created private tmux server,
  • And no claude -p or alternate direct transport is used.

Scenario S2 — Direct or fallback Claude transports are rejected

  • Given a skill, prompt, or script contains a runnable claude -p, claude --print, direct claude --permission-mode, direct prompt args, stdin piping into claude, input redirection such as claude < prompt.txt, unknown future forms such as claude --future-noninteractive x, prompt piping, raw interactive_shell(...claude...), process(...claude...), or os.execvp(...claude...) review path,
  • When the guardrail scans source or installed roots,
  • Then it fails unless the reference is a marked forbidden example or approved test fixture,
  • And it reports the offending path and line.

Scenario S3 — Auth works only in the real caller context

  • Given the current shell can authenticate Claude but an existing unrelated tmux session cannot,
  • When the launcher is run from the Pi prompt context or OpenCode subprocess context,
  • Then it creates its own private tmux server from that caller process,
  • And claude auth status is checked there as an early signal,
  • And the same-process smoke proves the real caller context is usable before relying on the launcher.

Scenario S4 — Auth failure fails closed with actionable evidence

  • Given claude auth status is false/nonzero or the TUI reports Not logged in,
  • When a smoke or review run starts,
  • Then the launcher exits nonzero with a classified auth failure,
  • And the output names what was tried, transcript path, tmux inspect command, and next user action,
  • And no alternate transport retry is attempted.

Scenario S5 — TUI readiness failure does not paste a prompt

  • Given interactive Claude starts but never reaches a usable input prompt,
  • When the readiness timeout expires,
  • Then the launcher fails as CLAUDE_TUI_NOT_READY,
  • And it preserves transcript and inspect command,
  • And it does not paste or submit the review prompt.

Scenario S6 — Prompt echo cannot produce false success

  • Given the pasted prompt wrapper contains both CLAUDE_REVIEW_ANSWER_START_<nonce> and the sentinel instructions,
  • When tmux captures prompt echo after Enter,
  • Then the launcher treats the pre-submit capture as diagnostics only,
  • And records a post-submit/post-clear boundary after the input box clears,
  • And ignores marker/sentinel text at or before that boundary,
  • And succeeds only on a marker/sentinel pair in Claude's answer region.

Scenario S7 — Boundary uncertainty fails closed

  • Given the launcher cannot prove the prompt echo has cleared before answer text appears,
  • When extracting the review output,
  • Then it fails as CLAUDE_TUI_BOUNDARY_UNCERTAIN,
  • And it preserves raw transcript and inspect metadata,
  • And it does not write a parseable success artifact.

Scenario S8 — OpenCode verdict parsing remains deterministic

  • Given OpenCode calls the launcher for a plan or implementation review,
  • When the launcher succeeds,
  • Then --output begins with ANSI-stripped, de-wrapped answer text strictly between the post-boundary answer-start marker and sentinel,
  • And linear_build_review.py parses verdicts only from that normalized answer region,
  • And raw pane transcript/TUI chrome cannot affect verdict parsing.

Scenario S9 — Installed fan-out matches source behavior

  • Given ./install.sh --pi and ./install.sh --opencode have run,
  • When installed prompts, commands, scripts, and skills are inspected and smoke-tested,
  • Then installed Pi and OpenCode paths resolve the same canonical launcher,
  • And installed roots contain no stale direct Claude transports.

Scenario S10 — Optional model-provider reviews cannot satisfy the required gate

  • Given a workflow mentions opencode-zen/claude-* or claude-opus-*,
  • When that workflow needs a required Claude Code review gate,
  • Then the model-provider review is classified as optional/non-gating,
  • And the required gate remains unsatisfied until the canonical Claude Code launcher succeeds.

P2 — Implement the canonical interactive tmux launcher

End State

skills/claude-code-review/scripts/claude_interactive_review.py satisfies P1 launcher tests and provides a single deterministic entrypoint for smoke and review modes.

Tests first

Run the P1 launcher test suite before implementation and confirm the relevant tests fail for the current stub/current behavior. During implementation, make the tests pass without weakening prompt-boundary, auth, readiness, or guardrail assertions.

Work

Implement locked CLI contract:

  • Normal review mode required inputs: --cwd <repo>, --prompt-file <path>, --output <path>, --review-name <slug>.
  • Normal review mode optional inputs: --timeout-seconds <n>, --sentinel <string>.
  • If --sentinel is omitted, the launcher generates a nonce-derived sentinel such as CLAUDE_REVIEW_DONE_<review-slug>_<pid>_<nonce> using the same collision-safe nonce source as the tmux socket.
  • Sentinel validation rejects empty strings, newline/control characters, strings shorter than a documented minimum length, strings containing shell metacharacter ambiguity that would affect prompt wrapper readability, and common/generic tokens such as DONE, END, or SUCCESS; invalid sentinels fail before tmux launch with a classified input-validation error.
  • Smoke mode required inputs: --smoke, --output <path>, --review-name <slug>.
  • Smoke mode optional inputs: --cwd <repo> defaulting to os.getcwd(), --timeout-seconds <n>.
  • Smoke mode has no --prompt-file; it runs same-process private-tmux auth and interactive TUI readiness only, without submitting a review prompt.
  • Smoke success exits 0 and writes/prints CLAUDE_REVIEW_SMOKE_READY plus private tmux socket/session metadata.
  • Smoke failure exits nonzero with the same classified missing-prerequisite/auth/TUI-readiness failure used by review mode.
  • There is deliberately no --verdict-regex; callers validate domain verdicts from output.

Implement tmux and Claude behavior:

  • Verify tmux exists.
  • Verify claude exists through zsh -ilc 'command -v claude'.
  • Create a private tmux server/socket named by a documented function of review slug plus pid/nonce.
